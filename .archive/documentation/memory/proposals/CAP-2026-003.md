## Current State

**Principle**: Code Quality - Linting, Formatting & Module Size (MANDATORY, v1.2.0)

Current relevant text (grandfathering clause and module size rules):

> **Non-Negotiable Rules (Module Size):**
> - New source files SHOULD NOT exceed 500 lines of code
> - New source files MUST NOT exceed 1,000 lines of code without documented justification
> - When modifying files that exceed 1,000 lines, contributors SHOULD extract changed logic into focused sub-modules where practical
> - **Grandfathered files** (exceeding thresholds at v1.2.0 ratification): `src/output/html.ts`, `src/core/analyzer.ts`, `src/types/index.ts`, `src/cli/commands.ts`, `src/core/daily-trends.ts` — these SHOULD be decomposed over time but are not blocking violations
>
> **Encouraged Practices:**
> - Split modules by single responsibility
> - Use barrel exports (`index.ts`) to maintain clean public APIs after decomposition
> - Target <300 lines per module for optimal reviewability
> - When a file exceeds 500 lines, create a tracking issue for decomposition

The current constitution encourages decomposition of grandfathered files but provides **no size guidance for the resulting decomposed modules** — nor does it address the scenario where a decomposition output itself exceeds 500 lines.

### Root Cause

The constitution mandates `<500 SHOULD NOT` for **new** files. However, it does not:
1. Clarify that a module resulting from decomposition of a grandfathered file is a "new file" subject to the full threshold
2. Set a **target size** for decomposition outputs (so teams can judge if a proposed split is sufficient)
3. Acknowledge that decomposition may sometimes require multiple successive splits to reach compliance (no "decomposition plan" pattern)

As a result, `html.ts` (grandfathered at ~2,500 lines) is being decomposed into sub-modules that are themselves 800+ lines — which is structurally better but still yields non-compliant new modules.

### Specific Examples

1. **`src/output/html-styles.ts`** (820 lines): Extracted from `html.ts` during v1.3.0 decomposition. Contains CSS template strings and style helpers. 820 lines exceeds the 500-line SHOULD NOT threshold by 64%. No tracking issue was created at extraction time per constitution guidance — because the guidance only triggers on existing files growing, not new files that start above threshold.

2. **`src/output/html-daily-trends.ts`** (817 lines, +69 growth): Extracted from `html.ts` during v1.3.0 decomposition. Contains 12 chart/table generation functions. Started large at extraction and has grown further. Tracking issue #6 was created reactively (post-audit).

3. **`src/index.ts`** (517 lines): Combines barrel exports with the `GitSpark` class implementation. Exceeds SHOULD NOT (500) by 3.4%. Constitution commentary notes three-layer architecture — `index.ts` as class+barrel hybrid violates single-responsibility guidance.

## Proposed Change

### Modification: Decomposition Sizing Rules

Add the following sub-clause to the **Non-Negotiable Rules (Module Size)** section:

---

**Decomposition Output Sizing:**
- A module created by extracting code from a grandfathered file is a **new file** subject to the full `<500 SHOULD NOT` threshold
- A decomposition extraction that produces a new file exceeding 500 lines **MUST** be accompanied by a follow-up decomposition tracking issue filed before the PR merges
- **Target size for decomposition outputs**: extracted modules SHOULD NOT exceed 300 lines at the time of initial extraction; if the extraction produces a module >300 lines, consider further splitting before merging
- Barrel files (`index.ts` pattern) **MUST NOT** contain business logic; if a barrel file exceeds 100 lines, the business logic must be extracted to a named module

**An additional grandfathering clause for pre-tracked decomposition results (at v1.3.0):**
- `src/output/html-styles.ts`: 820 lines — grandfathered pending tracking issue
- `src/output/html-daily-trends.ts`: 817 lines — tracked via issue #6 (in progress)

*(These two files existed before this amendment and are being tracked for remediation. They are not blocking violations but must not be allowed to grow further.)*

---

**Modify the Encouraged Practices section**:

Replace:
> - Target <300 lines per module for optimal reviewability
> - When a file exceeds 500 lines, create a tracking issue for decomposition

With:
> - Target <300 lines per module for optimal reviewability (both new modules and decomposition outputs)
> - When a file exceeds 500 lines, create a tracking issue for decomposition immediately — not reactively after an audit
> - When planning a decomposition, estimate extracted module sizes before coding; if any output will exceed 500 lines, plan successive splits up-front
> - Barrel files (`index.ts`) should be <50 lines; use them only for re-exporting, never for business logic

**Rationale**: The decomposition-as-solution pattern only delivers value if the resulting modules are meaningfully smaller. Without explicit output-size guidance, decomposition can replace one large file with two medium-large files — reducing violation count but not solving the underlying problem. Clear targets at extraction time prevent second-generation size debt.

**MUST Requirements:**
- New modules from decomposition tracked with an issue if they exceed 500 lines at creation
- Barrel files must not contain business logic
- `src/index.ts` business logic extracted to a dedicated module (follow-up task)

**SHOULD Recommendations:**
- Plan extraction sizes before starting decomposition
- Target <300 lines for newly extracted modules
- Use multiple sequential extractions when a single split would still produce large modules
