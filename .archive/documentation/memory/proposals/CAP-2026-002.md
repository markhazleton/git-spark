## Current State

**Principle**: Documentation Standards (MANDATORY)

Current relevant text (from constitution v1.2.0):

> **Non-Negotiable Rules (Code Documentation):**
> - All exported classes, interfaces, functions must have JSDoc comment blocks
> - Include `@param` for all parameters with descriptions
> - Include `@returns` for all return values with type information
> - Include `@example` for public APIs showing real usage
> - Document error conditions with `@throws`

The principle requires JSDoc on all exports but does **not** explicitly state who is responsible for adding JSDoc during module decomposition, or that JSDoc must follow extracted symbols to their new home.

### Root Cause

In two separate decomposition operations:

1. `html.ts` → `html-author-profiles.ts` (v1.2.x decomposition): Functions were extracted but JSDoc was not written at extraction time.
2. `html.ts` → `html-daily-trends.ts` (v1.3.0 decomposition): 12 functions extracted; none received JSDoc.
3. `index.ts` / `analyzer.ts` → `types/author.ts` (interface extraction): 8 interfaces moved without individual JSDoc.

The constitution principle is clear that JSDoc is required, but it provides no guidance for the **decomposition workflow** — making it easy to miss JSDoc as an implicit work item during module splitting.

## Proposed Change

### Clarification: Decomposition Documentation Obligation

Add the following subsection to the **Documentation Standards** principle under "Non-Negotiable Rules (Code Documentation)":

---

**Decomposition Documentation Obligation:**
- When extracting functions, classes, or interfaces into a new module (decomposition), the extractor **MUST** add or transfer complete JSDoc to each exported symbol before the PR is merged
- Decomposition tasks in `tasks.md` **MUST** include an explicit documentation sub-task: "Add JSDoc to all exported symbols in `<new-module>.ts`"
- A decomposition PR that introduces undocumented exports is a **Documentation Standards violation** regardless of whether the source module previously lacked documentation

**Guidance for Type Files (`src/types/**`):**
- Exported interfaces **MUST** have interface-level JSDoc describing purpose and usage context
- Properties that are non-obvious **SHOULD** have inline JSDoc (`/** description */`)
- Union/intersection types **MUST** have JSDoc explaining when each member applies

---

**Rationale**: Decomposition documentation obligation makes the responsibility explicit at the point where documentation debt is most commonly incurred. Without this guidance, developers reasonably treat "extract to module" as a code-only change and defer documentation — creating recurring audit findings of the same class.

**MUST Requirements:**
- JSDoc required on all exports at the time of decomposition PR merge
- Decomposition tasks must explicitly include a documentation sub-task

**SHOULD Recommendations:**
- Follow the `html-author-profiles.ts` pattern (successfully documented after DOC1) as a template
- Use `@example` showing how the function is called from its consuming module
- For type files, document the business meaning of each interface, not just its structure
